## 0. 一句话目标

**让 LLM 自己决定循环何时收敛**——acting agent 在执行中发现"信息不够"时，能主动触发新一轮 recon→refs→aggregator，整个外层闭环由 LLM 主导，工程层只提供"防止失控"的硬保底。

设计哲学与 v0.14.9 的 OUTPUT DEPTH（让 LLM 自己判断输出深度）完全一致——**不要在工程层硬性控制，把决策权交给 LLM**。

## 1. 背景与现状（必须读懂才能开始）

### 1.1 当前已有的两个循环（v0.14.13）

#### Path A — `@moa` chat participant（默认快速路径）

代码位置：[src/moaRunner.ts](../../src/moaRunner.ts) 的 `runP1Fanout()` 函数

```
sufficiencyLoop (maxReconRounds 默认 3)
├── Phase 0:   Recon agent（read-only 工具）
├── Phase 1:   N refs 并行（每个输出 {sufficient, missing, analysis}）
└── Phase 1.5: Sufficiency gate
               └── refs 多数说 sufficient=false → 收集 missing hints → 回到 Phase 0

[Phase 2]   Aggregator（一次性，不参与循环）
[Phase 3]   Acting agent（内部有 multi-turn tool loop，但外层无回路）
```

**关键缺陷**：Phase 3 之后**无任何反馈路径**。一旦 refs 说"够了"，就一路向下不回头——即使 acting agent 在改代码时发现"这个依赖我都没见过"，也没法回到 Phase 0。

> ✅ **已在 v0.15.0/v0.17.0 修复**：v0.15.0 实现了完整 5 角色 Hermes-style 闭环（见 §15），v0.17.0 把 `@moa` 切到迭代模式直到 Aggregator 自然收敛。

#### Path B — `#moa_orchestrate` LM tool（Hermes-style 完整闭环）

代码位置：[src/moaOrchestrator.ts](../../src/moaOrchestrator.ts)

```
MAX_ITER = 12
COMPLETENESS_THRESHOLD = 0.8
CONVERGENCE_WINDOW = 3

aggregator 输出 next_action:
  ├─ "recon_needed"        → 暂停，等 #moa_continue 喂回 subagent 结果
  ├─ "need_more_analysis"  → 再跑一轮（无新 recon）
  └─ "finalize"            → 退出循环
```

**关键缺陷**：**没有 acting agent**。这是纯"研究/分析"型循环，不能执行工具。状态持久化在 `<workspace>/.moa_cache/<task_id>/`。

> ✅ **已在 v0.15.0 修复**：v0.15.0 的 5 角色重构给 `#moa_orchestrate` 路径加了 Actor 角色（见 §15.1），与 Path A 共享同一套 moaCore 组件。

### 1.2 两条路径的根本割裂

| 维度 | Path A (`@moa`) | Path B (`#moa_orchestrate`) |
|---|---|---|
| 入口 | chat participant | LM tool |
| 有 acting agent？ | ✅ | ❌ |
| 完整闭环？ | ❌（只在 Phase 0-1.5 循环） | ✅（aggregator 主导） |
| 状态持久化 | recon dump 落盘但无 resume 能力 | ✅ 完整 state.json |
| 谁决策循环 | refs 的 `sufficient: bool` | aggregator 的 `next_action` |
| 用户体验 | 一句话发起 | 需要手动驱动 continue |

**v0.15.0 的核心任务**：把两条路径的优势合并——**给 Path A 加完整外层闭环，让 acting 也能触发新一轮 recon**。

> ✅ **实施状态**：v0.15.0 实际采用了"批次拆分"策略（见 §15.1），优先重构 `#moa_orchestrate` 为完整 5 角色闭环；原计划的"Path A 外层闭环"由 v0.17.0 的"@moa 迭代模式"以更简洁的方式实现（让 LLM-driven Aggregator 决定收敛，而非 dual-channel signaling + 硬保底）。原 §3.1 的 dual-channel 设计（moa_request_more_recon tool + `<MOA_STATUS>` 块）未实施，作为后备设计保留。

## 2. 设计原则（不可妥协）

### 2.1 LLM-driven control（核心哲学）

- **不要硬阈值**：不在工程层加"循环 3 次必须停"这种死规则
- **不要事后关键词检测**：不扫"cannot determine"这种启发式
- **让 LLM 输出结构化信号**：双信号通道（见 §3.1）

### 2.2 Hard guarantees（防止失控）

LLM 主导 ≠ 完全放任。必须有"防止跑飞"的硬保底：

| 保底类型 | 默认值 | 配置项 | 触发动作 |
|---|---|---|---|
| 总字符预算 | 500K chars | `moa.maxOuterLoopChars` | 强制 CONVERGED |
| 硬循环上限 | 5 轮 | `moa.maxOuterLoops` | 强制 CONVERGED |
| 收敛检测 | cosine sim > 0.8 | `moa.convergenceSimilarity` | 强制 CONVERGED |
| 单轮 token | 各模型自定 | 已有 | 报错 fallback |

### 2.3 Backward compatibility（向后兼容）

- **默认行为不变**：`moa.maxOuterLoops` 默认 1（即不循环），与 v0.14.x 行为完全一致
- 用户**主动开启**循环：`settings.json` 配置 `moa.maxOuterLoops: 5`，或用 `@moa --deep-dive` 命令
- 已有的 `sufficiencyLoop`（Phase 0-1.5）**保留不动**，只是新外层闭环包住它

### 2.4 配置而非硬编码

所有阈值都是 `moa.*` 配置项，便于用户根据模型容量调整（GLM-5.2 1M context 可以放宽，DeepSeek-V4-Flash 64K 要收紧）。

## 3. 核心设计

### 3.1 双信号通道（Dual-channel signaling）

**为什么双保险**：单一通道都有失败模式——纯 tool 依赖 LLM 会用工具，纯 XML 块依赖 LLM 遵守格式。两者都识别 = 纵深防御。

#### Channel A — `moa_request_more_recon` LM tool

新注册一个工具：

```typescript
// 伪代码
vscode.lm.registerTool('moa_request_more_recon', {
  description: 'Signal that the acting agent needs more information before continuing. '
             + 'Use ONLY when genuinely blocked — not for "nice to have" context.',
  inputSchema: {
    type: 'object',
    properties: {
      missing_hints: {
        type: 'array',
        items: { type: 'string' },
        description: 'Specific items needed (file paths, symbols, paper IDs, ...)',
        minItems: 1,
        maxItems: 10,
      },
      rationale: {
        type: 'string',
        description: 'Why this info is critical (1-2 sentences)',
      },
    },
    required: ['missing_hints', 'rationale'],
  },
  async invoke({ input, toolInvocationToken }, token) {
    // 把信号写入 toolInvocationToken 关联的 outer-loop state
    // 然后抛特殊异常或返回特殊 marker 让 acting agent 退出当前 multi-turn loop
    return {
      content: [new vscode.LanguageModelTextPart(
        '[MoA outer-loop] Signal received: requesting recon round N+1. '
        + 'Exiting current acting loop; will reconvene after new recon pass.'
      )]
    };
  }
});
```

**关键设计点**：
- `missing_hints` 是**必填**且 minItems=1，强制 LLM 想清楚到底缺什么（防止"我感觉信息不够"这种空喊）
- `rationale` 必填，让 LLM 解释为什么（自我审视，减少误触发）
- 返回的 TextPart 是给 acting agent 看的"已收到信号，准备退出"提示
- 该工具**只对 acting agent 可见**（recon 是 read-only 模式不暴露）

#### Channel B — `<MOA_STATUS>` 结构化输出块

要求 acting agent 输出末尾附 XML 块（prompt 工程，无 tool overhead）：

```
<MOA_STATUS>
STATE: CONVERGED | NEEDS_MORE_RECON
MISSING_HINTS:
  - src/auth/session.ts 的 token 过期逻辑
  - 用户期望的认证方式（JWT vs session）
RATIONALE: 当前代码看起来用了双重 session 机制，但具体是 JWT 还是 session-based
           不读代码无法决定。继续 acting 会基于错误假设。
</MOA_STATUS>
```

**解析规则**：
- 优先识别 tool call（Channel A）—— 如果 acting 调了 `moa_request_more_recon`，以 tool 信号为准
- 没有 tool call → 扫输出末尾找 `<MOA_STATUS>` 块
- 两个都没有 → 默认 CONVERGED（防止 LLM 忘了输出）

### 3.2 完整闭环数据流

```
                ┌─→ [outer loop N, maxOuterLoops=5] ──────────────────┐
                │                                                     │
                │   ┌── sufficiencyLoop (已有, maxReconRounds=3) ──┐  │
                │   │   Phase 0  Recon                               │  │
                │   │   Phase 1  N refs                              │  │
                │   │   Phase 1.5 Sufficiency gate                   │  │
                │   └───────────────────────────────────────────────┘  │
                │   ↓                                                  │
                │   Phase 2  Aggregator                                │
                │   ↓                                                  │
                │   Phase 3  Acting agent (multi-turn tool loop)       │
                │   │                                                  │
                │   ├─ [CONVERGED] ──────────→ 退出外层，输出最终结果  │
                │   │                                                  │
                │   └─ [NEEDS_MORE_RECON] ───→ 收集 missing_hints     │
                │                              写入 outer_loop_state   │
                │                              回到外层循环顶部 ↑     │
                │                                                     │
                └─────────────────────────────────────────────────────┘

硬保底检查（每轮外层结束后）：
  - outerLoopCount >= maxOuterLoops?     → 强制 CONVERGED
  - totalCharsConsumed >= maxOuterLoopChars? → 强制 CONVERGED
  - cosineSimilarity(missing_hints_N, missing_hints_N-1) > 0.8? → 强制 CONVERGED
```

### 3.3 状态持久化

新增 outer-loop 状态文件：

```
<workspace>/.moa_cache/outer_loops/<task_sha>/
├── outer_state.json          # 外层循环状态
│   {
│     "task_sha": "...",
│     "prompt": "...",
│     "started_at": "...",
│     "outer_loops_used": 2,
│     "max_outer_loops": 5,
│     "total_chars_consumed": 152340,
│     "max_chars": 500000,
│     "missing_hints_history": [
│       ["src/foo.ts", "bar() usage"],
│       ["src/foo.ts", "bar() usage"]   # 第二轮重复 → 触发收敛检测
│     ],
│     "last_status": "NEEDS_MORE_RECON",
│     "converged_reason": null  # 或 "max_loops_reached" / "budget_exhausted" / "convergence_detected"
│   }
├── loop_001/                 # 第一轮外层
│   ├── recon_summary.md      # (复用现有 recon dump 机制)
│   ├── ref_outputs/          # (复用)
│   ├── aggregator.md         # (复用)
│   ├── acting_final.md       # acting 这一轮的最终输出
│   └── signal.json           # 信号记录：tool call 或 <MOA_STATUS> 解析结果
├── loop_002/
│   └── ...
└── final.md                  # 最终用户可见输出
```

**复用 v0.14.3 的 recon dump 机制**，只是在外面包一层 outer_loops/ 目录。

## 4. 配置项新增

全部在 `moa.*` namespace 下：

| Key | Type | Default | Description |
|---|---|---|---|
| `moa.maxOuterLoops` | number | `1` | 外层闭环最大轮数。1 = 当前行为（不循环）。5 = 最多 5 轮 acting→recon。 |
| `moa.maxOuterLoopChars` | number | `500000` | 外层闭环总字符预算（所有轮 recon+ref+acting 输出累计）。超过强制 CONVERGED。 |
| `moa.convergenceSimilarity` | number | `0.8` | missing_hints 与上一轮的相似度阈值，超过视为"LLM 在重复"，强制 CONVERGED。 |
| `moa.enableMoaSignalTool` | boolean | `true` | 是否注册 `moa_request_more_recon` LM tool。关闭则只靠 `<MOA_STATUS>` 块。 |
| `moa.outerLoopDeepDiveCommand` | boolean | `true` | 启用 `@moa /deep-dive <question>` chat 命令（覆盖 maxOuterLoops 到 5）。 |

     const signal = parseMoaStatusBlock(result.output);
     if (!signal || signal.state === 'CONVERGED') break;

## 6. 关键实现细节

### 6.1 Acting agent system prompt 补丁

现有 acting prompt（[src/actingAgent.ts](../../src/actingAgent.ts) L357-390）末尾加：

```
=== MOA OUTER-LOOP SIGNALING (v0.15.0+) ===

You are running inside a closed-loop MoA pipeline. If `moa.maxOuterLoops > 1`,
your output is NOT necessarily final — you can request another round of
recon + ref + aggregator analysis if you discover the current task cannot
be completed without new information.

WHEN to signal NEEDS_MORE_RECON (be strict):
  - You would have to GUESS to continue, and a wrong guess would mislead
    the user (e.g. "is this JWT or session-based auth?" — you genuinely
    cannot tell without reading more code)
  - A specific file / symbol / external reference would UNBLOCK you, and
    you can NAME it concretely (not "more context about authentication"
    but "src/auth/session.ts:token rotation logic")
  - You've already tried to find it via your own tool calls and failed

WHEN NOT to signal (default behavior, no signal needed):
  - You have enough info to produce a useful answer (even if imperfect)
  - The "missing" info is nice-to-have but your analysis stands without it
  - You're just uncertain about your conclusion — that's normal, state it
    in the answer and CONVERGE

HOW to signal (pick ONE):
  Option A (preferred): Call the `moa_request_more_recon` tool with
    missing_hints (specific items) + rationale.
  Option B (fallback if tool not available): Append to your output:

    <MOA_STATUS>
    STATE: NEEDS_MORE_RECON
    MISSING_HINTS:
      - <specific item 1>
      - <specific item 2>
    RATIONALE: <1-2 sentences why this is critical>
    </MOA_STATUS>

If neither applies, just end your response normally (implicit CONVERGED).

REMEMBER: each outer loop costs the user tokens. Be honest about whether
you really need more info vs. you're just being thorough.
```

### 6.2 missing_hints 相似度计算

简单 Jaccard 即可（不需要 embedding）：

```typescript
function jaccardSimilarity(a: string[], b: string[]): number {
  // 把每个 hint 规范化：小写、去标点、按词分
  const normalize = (s: string) => s.toLowerCase().replace(/[^\w\s]/g, '').split(/\s+/).filter(Boolean);
  const setA = new Set(a.flatMap(normalize));
  const setB = new Set(b.flatMap(normalize));
  const intersection = [...setA].filter(x => setB.has(x)).length;
  const union = new Set([...setA, ...setB]).size;
  return union === 0 ? 0 : intersection / union;
}
```

**为什么 Jaccard 够用**：missing_hints 是短的文件路径/符号名，词级重叠就是好信号。如果两轮都问"src/foo.ts 的 bar()"，Jaccard 一定高。

### 6.3 toolInvocationToken 与 outer-loop state 的关联

**问题**：`moa_request_more_recon` 工具被调用时怎么知道是哪个 outer loop？

**解法**：用 `toolInvocationToken` 做 key。在 `extension.ts` 维护一个 module-level Map：

```typescript
// extension.ts 或新模块 outerLoopStateRegistry.ts
const outerLoopStates = new WeakMap<vscode.ChatParticipantToolToken, OuterLoopState>();

export function registerOuterLoop(token: vscode.ChatParticipantToolToken, state: OuterLoopState) {
  outerLoopStates.set(token, state);
}

export function getOuterLoop(token: vscode.ChatParticipantToolToken): OuterLoopState | undefined {
  return outerLoopStates.get(token);
}
```

`WeakMap` 让 token 被 GC 时 state 自动清理，避免内存泄漏。

### 6.4 Acting agent loop 如何响应 tool call

当 acting agent 调用 `moa_request_more_recon` 时，需要在 `runActingAgent` 的 tool-call 循环里捕获信号并优雅退出：

```typescript
// 在 runActingAgent 内
let outerLoopSignal: OuterLoopSignal | null = null;

for (const call of toolCalls) {
  if (call.name === 'moa_request_more_recon') {
    // 解析 input
    outerLoopSignal = {
      missingHints: call.input.missing_hints,
      rationale: call.input.rationale,
      source: 'tool_call',
    };
    // 不实际执行（不需要），直接构造一个 TextPart result
    continue;
  }
  // ...其他 tool 正常处理
}

// 循环外检查
if (outerLoopSignal) {
  return {
    output: finalOutput,
    outerLoopSignal,
    iterations,
    // ...
  };
}
```

| 简单事实 | "2+2 等于几？" | 1 | CONVERGED |
| 简单代码查询 | "package.json 的 version 字段是多少？" | 1 | CONVERGED |
| 多文件重构 | "把这个项目的所有 callback 改成 async/await" | 2-3 | NEEDS_MORE_RECON at least once |
| 开放研究 | "分析冬眠动物的代谢调控机制" | 1-2 | CONVERGED（refs 训练知识足够） |
| 故意刁难 | "修改 src/foo.ts 的 bar() 函数"（但 foo.ts 不存在） | 2 | NEEDS_MORE_RECON 第一次，CONVERGED 第二次 |

### 7.3 性能基准

同一问题在 v0.14.13 vs v0.15.0 的：
- 总 token 消耗（输入+输出）
- 总墙钟时间
- 调用的模型次数

**期望**：简单问题无显著差异（<5%）；复杂问题 v0.15 可能多花 30-100% token 但答案质量明显更好。

## 12. 开放问题（实施前需决定）

### Q1：`maxOuterLoops=1` 时是否还注册 `moa_request_more_recon` tool？

- 选项 A：不注册（节省 tool slot，LLM 看不到也调不了）
- 选项 B：注册但 invoke 时返回 "outer loop is disabled, treat as CONVERGED"
- **建议**：选项 A（更干净，行为一致）

### Q2：外层循环的进度是否进入 chat history？

- 进入：用户后续 @moa 能引用"上一轮你说了 X"
- 不进入：保持 Hermes 设计（ref/acting 输出是 side-channel）
- **建议**：不进入（与 v0.14 thinking-mode 一致），但 OutputChannel 完整记录

### Q3：deep-dive 命令是否持久化偏好？

- 用户 `@moa /deep-dive` 一次后，后续 `@moa` 是否也变 deep-dive？
- **建议**：不持久化（每次都要显式 /deep-dive），避免意外

### Q4：相似度检测误判怎么办？

- LLM 真的需要两轮问相似但不完全相同的东西（如 "foo.ts 的 bar()" vs "foo.ts 的 baz()"），Jaccard 可能误判收敛
- **建议**：相似度阈值默认 0.8 偏保守（误判方向是"该继续但停了"而不是"该停但继续"）；用户可调

### Q5：是否给 `#moa_orchestrate` 路径也加 acting agent？

- 这是更大的工作（统一两条路径）
- **建议**：v0.15.0 不做（scope creep 风险），作为 v0.16.0 的目标

### 15.2 hotfix 1：Actor evidence 提取崩溃

**症状**：`moa_continue` 报 `Iteration failed: Cannot read properties of undefined (reading 'length')`。第 2 轮文件完整落盘到 `iteration_002/`（含 `actor_result.json`），但 `state.history` 未 push → `state.json` 仍是 iteration=1 → `timeline.md` 漏写 iteration_002。

**根因**：[src/moaOrchestrator.ts](../../src/moaOrchestrator.ts) 原 evidence 提取循环直接访问 `ar.action.content.length`。Actor LLM 经常只回 `type/target/rationale`，不回 `content`（因为对 Actor 来说 `content` 字段无意义，它报告的是执行结果摘要而非 action 原文）。

**修复**：
1. 抽出纯函数 `buildActorEvidence(ar, iterNum)` 到独立文件 [src/moaCore/actorEvidence.ts](../../src/moaCore/actorEvidence.ts)（独立文件便于单测，避免 import vscode 的 mock 问题）
2. 用 `ar.action?.content` + `typeof string` 检查；缺失时用 `artifacts` + `output_chars` + `rationale` 构造兜底 snippet
3. 测试覆盖：`test/hotfix.test.ts`（8 cases，含复刻真实崩溃数据形状的回归保护）

**为什么抽到独立文件**：moaOrchestrator.ts 顶部 `import * as vscode from 'vscode'` 在 Node test env 无法直接 import。把纯函数抽到不依赖 vscode 的文件，测试只需 `import { buildActorEvidence } from '../src/moaCore/actorEvidence'`，零 mock。

### 15.3 hotfix 2：自动 finalize 时 `final.md` 缺失

**症状**：`runIteration` 在 `aggOutput.next_action === 'finalize'` 时直接设 `state.status = 'finalized'` 返回，但**未调用 `finalizeTask`**。用户看到 status=finalized 但 `final.md` / `final.json` 不存在，需要手动再调一次 `#moa_finalize`。

**根因**：`finalizeTask`（生成 final.md + 调 aggregator 跑 summary + 写 final.json）只在 `#moa_finalize` 工具入口被调用。runIteration 的三个 finalize 分支（MAX_ITER / `next_action='finalize'` / `shouldStop`）都没触发它。

**修复**：在 `runIteration` 末尾 `return state;` 之前，检查 `state.status === 'finalized'`，是的话调 `finalizeTask(taskId, token)` 自动收尾。失败用 `progress?.()` 报警告但不阻塞返回。

## 附录 B：设计决策记录（ADR-style）

### ADR-001：信号机制选 dual-channel 而非单 channel

- **决策**：tool call + XML 块双保险
- **理由**：单 tool 依赖 LLM 学过用工具（弱模型不会用）；单 XML 块依赖 LLM 遵守格式（输出会变形）。双保险覆盖更多模型。
- **代价**：实现复杂度翻倍（两个代码路径），维护成本高
- **反转条件**：实测发现某一通道从未被触发（如所有主流模型都更倾向 tool call），可以下线另一通道

### ADR-002：默认 maxOuterLoops=1（向后兼容）

- **决策**：默认行为与 v0.14.x 完全一致
- **理由**：避免对现有用户造成意外 token 消耗；循环是 opt-in 而非 opt-out
- **代价**：新功能默认不启用，用户需要发现 `--deep-dive` 命令或配置项
- **反转条件**：v0.16.0 后视用户反馈考虑把默认改为 2 或 3

### ADR-003：硬保底用 Jaccard 而非 embedding

- **决策**：词级 Jaccard 相似度（无外部依赖）
- **理由**：missing_hints 是短字符串（文件路径、符号名），词级重叠就是好信号；不引入 tokenizer/embedding 依赖（保持 bundle 小）
- **代价**：语义相似但用词不同的 hints 会漏检（"user authentication" vs "login system"）
- **反转条件**：实测发现误判率高，再考虑加 embedding
