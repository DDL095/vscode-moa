## §1 本次修订已完成事项（5 类）

### P0 — Actor gate 顺序 bug 修复（核心）

**根因定位**（详见 [`src/moaOrchestrator.ts`](../../src/moaOrchestrator.ts) 阶段 5 Gate）：

原顺序：
```
1. iteration >= MAX_ITER         → max_iter 强制收敛
2. next_action === 'finalize'    → 自然收敛
3. shouldStop(state)             → should_stop 强制收敛   ← bug 在这
4. next_action === 'actor_needed'→ 跑 Actor              ← 永远到不了
5. else                          → recon_needed
```

`shouldStop()` 检测条件：连续 `CONVERGENCE_WINDOW=3` 轮 completeness 不变。

**死亡螺旋场景**：

| 轮次 | Aggregator 建议 | 实际发生 | completeness |
|---|---|---|---|
| 1 | `actor_needed` | 窗口未满，放行 | 0.15 |
| 2 | `actor_needed` | 窗口未满，放行 | 0.72 |
| 3 | `actor_needed` | 窗口满 3，检查 Δ | 0.65（**降了**）|
| 4-10 | `actor_needed` | shouldStop 触发 → 强制 finalize | 卡 0.90-0.95 |

**修复方案**（A 方案，最小改动）：调换 gate 顺序

```
1. iteration >= MAX_ITER         → max_iter 强制收敛
2. next_action === 'finalize'    → 自然收敛
3. next_action === 'actor_needed'→ 跑 Actor              ← 提前
4. shouldStop(state)             → should_stop 强制收敛   ← 移到 actor 之后
5. else                          → recon_needed
```

**为什么 B 方案（shouldStop 豁免）次优**：B 方案需要在 shouldStop 函数里加
`if (lastNextAction === 'actor_needed') return false`，但这破坏了 shouldStop
作为"收敛检测器"的语义纯洁性。A 方案的 gate 顺序更符合角色优先级
（Actor 优先于收敛检测）。

**测试**：v0.18.4 发布后用相同任务（`整理 moa 插件...`）重跑，预期 Actor 列
应该至少有 1 轮 `1/1` 或 `N/N`。

## §4 教训（Lessons Learned）

### L1 — Gate 顺序必须与角色优先级一致

任何"先判断收敛、后判断执行"的 gate 顺序都会导致执行型角色空跑。
原则：**收敛检测永远在执行型角色之后**，否则收敛检测会把"未执行"误判为"停滞"。

### L2 — 版本号是单一真相源，必须动态化

硬编码版本号在 3 次发版后必然漂移（v0.18.3 修复了 v0.18.2 的漂移，
v0.18.4 又发现新的漂移点）。**根治方案是动态化**，不是"下次发版时仔细检查"。

### L3 — MoA 自审报告的 Action Items 必须人工执行

Actor 角色空跑的根本问题意味着 MoA 自身**无法自动执行**修订建议，
必须有主会话（人或 LLM 主对话）按 Action Items 手工落地。
这是 v0.18.4 的 Actor 修复之后才能改变的限制。
