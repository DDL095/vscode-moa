## 背景与诊断结论（iteration 6 验证后）

通过对 src/actingAgent.ts、src/moaCore/actorEvidence.ts、长程路线图附录 C 的逐文件读取 + 对照 v0.18.4 路线图，本轮 MoA 收敛到以下硬结论：

### 结论 1：actorEvidence.ts 守卫已就位，无 undefined.content 风险

src/moaCore/actorEvidence.ts:48 已使用 typeof rawContent === string && rawContent.length > 0 做显式守卫，并在缺失时回退到 artifacts + output_chars + rationale 合成 snippet。iter2 的 gate 守卫缺失假说不成立。III-1 的启用不会因此崩溃。

### 结论 2：21 次 tool_calls → 0 success action 的根因不在 actingAgent 内

src/actingAgent.ts 的循环结构（lines 478-740）逻辑闭环：

- iterations < maxIter（maxIter = 12 acting / 50 recon）— 12 是上限，21 次超过 acting 上限触发 hitIterationCap
- toolCallsSucceeded 与 toolCallsFailed 在 try/catch 内自增（line 663 / line 695）— 不依赖 Actor schema
- 真正的 actor schema 解析发生在 moaCore/roles.ts 的 parseActorActions，需独立排查

→ 零执行的真正根因在 Actor LLM 返回的 JSON schema 不被 parser 接受，不是 actingAgent.ts 的循环 bug。

### 结论 3：用户编号 19/21/22 ↔ 路线图项目映射

| 用户原话编号 | 内容摘要 | 对应路线图项目 | 阶段 |
|---|---|---|---|
| 19 | 让 llm 拉全工具 subagent | III-3（依赖 III-2） | v0.21.x |
| 21 | 删除 bak 备份 + 清单落盘 | III-2 子项 | v0.21.x |
| 22 | 汇总让用户决定清理 | III-2 子项 | v0.21.x |

用户要求的 19/21/22 全部依赖 III-2（保守执行模式），而 III-2 又依赖 III-1（开关+可见化）。因此 v0.19.0 的最高杠杆动作是先把 III-1 快赢落地。

### 结论 4：vscode.lm.tools 128 工具硬限制

actingAgent.ts:200 的 getActingTools 透传 vscode.lm.tools 全部列表——这意味着一旦 III-3 把全工具暴露给 Actor，工具数量将受 vscode.lm.tools 上限约束（实测 ~128）。需在 III-3 实施前验证规避方案（research_more 已派发）。

## P1 — III-1 快赢：enableActingAgent 运行时接线

背景：package.json 已声明 moa.enableActingAgent 配置项（line 456 一带），但 src/moaOrchestrator.ts 的 actor_needed gate 未读取该值。启用会让 Actor 自动执行，用户在 chat 中看不到过程。

目标：当 moa.enableActingAgent === false 时，actor_needed 分支降级为 recon_needed，不执行 Actor；为 true 时正常执行。

步骤：

1. 配置读取层：在 src/moaConfig.ts 添加 getEnableActingAgent(): boolean 工具函数（含默认值 false + workspace scope + global scope fallback）
2. 运行时接线：在 src/moaOrchestrator.ts 的 actor_needed 分支前置守卫（伪代码见 long-term-roadmap.md III-1）
3. NLS 配套：在 package.nls.json 与 package.nls.zh-cn.json 中补全 moa.enableActingAgent 的 description
4. 单元测试：mock moaConfig.getEnableActingAgent() 返回 false，验证 gate 降级路径

验收：

- moa.enableActingAgent=false 时 .moa_cache/<task_id>/ 不再生成 iteration_N/actor.json
- 配置变更后无需重启 VSCode 即生效
- 中文 / 英文 VSCode 双语显示正常

工作量：1-2h

## P2 — 缓存生命周期管理（新增条目）

问题：用户问每个对话与 moa 的任务缓存是怎么管理的——现有 .moa_cache/<task_id>/ 无 TTL、无容量上限、无多工作区命名空间隔离。

目标：在 src/moaConfig.ts 添加配置组 MoaCacheConfig（ttlDays / maxTasksPerWorkspace / maxTotalSizeMB / cleanupStrategy）。

步骤：

1. package.json 新增四个 moa.cache.* 配置项
2. NLS 双语：补全 4 个 description
3. cleanup 模块：新增 src/cacheCleanup.ts，启动时扫描 .moa_cache/：按 task_id mtime 排序；超过 TTL → 删除；超过 maxTasksPerWorkspace → 删除最早 N 个；超过 maxTotalSizeMB → 按 cleanupStrategy 删到阈值内
4. 命名空间隔离：在 .moa_cache/ 下按 workspace_fingerprint (hash(workspaceFolders[0].fsPath)) 建子目录，避免多 workspace 串扰
5. 执行时机：VSCode 启动后 5s 异步执行 + 每次任务完成后异步检查

验收：

- .moa_cache/<workspace_fingerprint>/<task_id>/ 路径生效
- 启动后 .moa_cache/ 总大小不超过 maxTotalSizeMB
- 删除前在 OutputChannel 打 manifest

工作量：4h

## P3 — III-3 方案修正：工具子集动态筛选 + isReadOnlyTool 重构

问题：原 III-3 方案直接传 vscode.lm.tools 全量——但 128 工具硬限制 + 重叠命名空间会导致 LLM 选错工具。isReadOnlyTool 当前按工具名前缀正则匹配（actingAgent.ts:140-170），扩展性差。

目标：把全量工具改为动态子集——按 task 类型 + role 类型动态裁剪。

步骤：

1. task 类型识别：在 Planner 输出中追加 task_type，由 actor_needed 分支读取
2. 工具分类标签：为每个 vscode.lm.tools 加 category 标签
3. isReadOnlyTool 重构：从正则前缀改为 category + action intent 双维度
4. 动态子集算法：按 category 优先级过滤 → 按 description 关键词匹配 → 截断到 maxCount
5. 规避 128 限制：分批注入或多轮 tool registration，待 research_more 结果决定

验收：

- 工具数量从 ~150 降到 ≤64
- code 类型任务不暴露 web search / ocr 工具
- research 类型任务不暴露 file editing 工具

工作量：3-4h（不含 128 限制规避方案研究）

依赖：P0 根因诊断完成 + research_more 结果

## P4 — I-2 实时推送 EventEmitter 通道设计

问题：用户要求实时查看相应的细节与内容——当前 5 个 OutputChannel 是 fire-and-forget，无订阅机制。

目标：在 src/moaOrchestrator.ts 暴露 EventEmitter，订阅者可监听 task:started / iter:started / role:invoked / role:completed / actor:action_started / actor:action_completed / task:finalized 等事件。

步骤：

1. 在 MoaOrchestrator 类中添加 _emitter 与 on<K> 公共 API
2. 在每个 callPlanner / callRecon / callRefs / callAggregator / callActor 钩子点 emit 对应事件
3. 在 src/webviewDashboard.ts（II-1）中订阅这些事件，实时刷新 Webview
4. 在 5 个 OutputChannel 的 appendLine 处也 emit 对应事件（统一入口）

验收：

- 外部扩展可以 vscode.extensions.getExtension(moa).exports.on(role:invoked, ...) 订阅
- Webview 在 actor 执行时实时刷新

工作量：4-6h

依赖：P3 + II-1

## P5 — I-2 扩展字段映射角色提交信息汇报

问题：用户问每个角色的提交信息都有汇报么——当前每个角色只在 OutputChannel 打进度，无结构化提交摘要。

目标：在 I-2 的 meta.json 扩充方案中，新增 per_role_submissions[] 字段（详见 long-term-roadmap.md I-3）。

验收：
